You must be familiar with those short, weird links, which, upon clicking, can either take you to your favourite blog (this one), or a scammy website, or some darker corner of the internet. They are typically called short URLs, where a service takes your actual URL (often called a long URL) and provides you with a shorter version. Internally, the service keeps a mapping of short URL to long URL. Therefore, when you visit that short URL, you are actually making a request to that service, which resolves that short URL to the corresponding long URL, and then redirects you to the resolved long URL. It often works so quickly that you hardly notice the redirect. Have you ever wondered how that short URL gets generated? The key observation is that, regardless of how it is generated, it must be unique. No two different long URLs should share or point to the same short URL (often called collisions), or some future Tuesday becomes much more annoying than originally planned. How can we honour such a fragile and sacred rule? Let's explore a few solutions.


## Solution 1: Short URLs from random data/bytes

This one is extremely basic in theory. You just ask your system to generate a totally random value of a certain length, and that's your short URL. System (your machine + your environment + your favourite programming language) talks in bits and bytes. So asking for a random value of a certain length N is usually asking the system to give you N random bytes, which you can then encode in a different numeric system like decimal, hex, base62, etc. This solution is very easy to implement, but it comes with a few challenges.

### Challenges

- **Needs good system entropy**: Entropy in itself is a lengthy topic, but in simpler words, it means it's a measure of unpredictability. A high entropy system yields more unpredictable (or random) values, whereas a low entropy system results in the opposite. We need high entropy to minimise collisions in the generated short URLs. In such a system, collisions can occur, even if the probability is extremely low, because the system is probabilistic. There will always be a chance that the next randomly generated value is the same as the one generated some time ago.
- **Choosing the right length is not optional**: The length of the random value decides the output space. A 4-character base62 short URL gives you 62^4, roughly 14.7 million possible values. A 6-character base62 short URL gives you roughly 56.8 billion possible values, which is much better, but still needs to be evaluated against expected scale, retry tolerance, and abuse patterns. The question is not just "how many URLs can we theoretically fit?", but "how likely are collisions before we reach that number?"
- **Higher database operations**: Because there is always a chance of a collision, you have to enforce a unique constraint in your database. One way to do this is to generate a value, check the database to ensure it does not already exist, and then insert it. That sounds reasonable, but it has a race condition. If two services generate the same short URL at roughly the same time, both can check the database, both can see nothing, and both can think they are safe. They are not safe. They are just confidently wrong at the same time. The safer pattern is usually: generate candidate, insert, handle unique constraint violation, retry. Initially, collisions are rare, so this keeps latency low. But as more URLs are generated, collisions become more likely, retries increase, and latency gets worse.
- **Deduplication is harder**: This might not be much of an issue, but if deduplication is a goal, then it's tedious to achieve because each new generation results in a new short URL even if the long URL remains the same. The only way to achieve deduplication is to perform a reverse lookup in the database: given a long URL, find a short URL and return it. This can introduce latency (a trip to the database), and you might also need to maintain an additional database index for those reverse lookups only to serve a small number of deduplication queries. However, as I said, it's only an issue if deduplication is required. Otherwise, this can be safely ignored; it's not like the world is ending on a Monday.
- **Custom aliases and reserved paths need protection**: If your application has routes like `/api`, `/admin`, `/login`, or allows users to choose their own aliases, the random generator must not accidentally produce one of those values. You can solve this by keeping generated URLs in a separate namespace, reserving certain words, or checking against a blocklist before insertion in the database.

## Solution 2: Short URLs from hash functions

Hash functions have a special property: given arbitrary-length input data, they deterministically map it to a fixed-length output. Same input, same output. Different input, usually different output, but not always. There is always a chance of collision simply because there is a finite output space, and two inputs can map to the same output. Depending upon the size of that output space, the chances are usually low enough that we mostly ignore them. For example, a very common hash function is SHA-256. The outputs are always 256 bits long, and there can be roughly 1.16 × 10^77 possible outputs. For reference, there are an estimated 10^80 atoms in the observable universe. Imagine how much computing power is required to find a collision. Although theories like the [birthday paradox](https://en.wikipedia.org/wiki/Birthday_problem) increase the chance of finding a collision, the chances still remain extremely low for any existing practical system.

Enough chit-chat about the hash functions. Let's explore how they serve as a viable solution for our problem. It's pretty obvious, actually: we pass our long URL through a hash function and get an output we can use in our finalised short URL. You see, I did not say we use the output as a short URL; technically, we can, but we usually don't. The reason is that the output is usually long enough to defeat the purpose of the short URL in the first place. For example, if we use SHA-256, we get exactly 256 bits as output. That's 32 raw bytes, 64 characters in hex, and usually around 43 characters in base64/base62-style encodings. Not a good way to represent a short URL, which in fact might be longer than the original long URL.

So what do we do? Well, one way is to encode the output of a hash function in base62 and then take N characters from it. Why base62 and not the popular cousin base64? Well, base64 also includes characters like `+` and `/`. Those characters usually have special meaning in a URL; `+` might represent spaces, and `/` might represent a URL segment, and both of them can mess up our short URL. base62, on the other hand, only has 26 letters [a-z], 26 letters [A-Z], and 10 digits [0-9]. Taking N characters from the base62-encoded output helps us shorten the original output, making it suitable for use as a short URL.

N is now important, as it controls the output space (the number of possible unique short URLs) and hence the likelihood of collisions. For example, if N = 8, then there are 62 ^ 8 = approx. 218 trillion possible unique outputs. A short URL with just 8 characters is still compact enough. A very nice byproduct of this solution emerges from the deterministic nature of the hash functions: same input -> same output, which means we can very easily achieve deduplication. The same long URL always produces the same short URL. Hash function solution is great, and one of the things you should try out, but it has its own challenges.

### Challenges

- **Collision control**: I know, I know. You might be thinking, if good hash functions practically have almost zero chances of collisions, then what's this fuss about? Well, remember, we encoded the output in base62 and then took N characters. That N is the root of this challenge. Set it low, you get nice short URLs, but you increase the collision chances; set it high, and you get longer short URLs but lower chances of collisions. If `S` is the total output space (`62^N`) and `n` is the number of short URLs already generated, the chance that the next generated short URL collides with an existing one is approximately `n / S`.

    Let's assume N = 8 and n = 10 billion. Now, the chance that a newly generated short URL will collide with an existing one is approximately 10 billion / 62^8 = approx. 0.00458%. That's about 1 collision for roughly every 22K URLs generated. That's the chance for the next URL. If we ask a slightly different question, "how many collisions should we expect after generating 10 billion short URLs?", the birthday-bound estimate is `n(n - 1) / 2S`, which gives us roughly 229K expected collisions. In other words, one collision is unlikely, but after billions of generated URLs, there are billions upon billions of pairs that could collide. Not ideal. The worst part is that the collision probability keeps increasing as more and more URLs are generated, unless we increase the value of N, which lowers the probability but does not make the situation entirely go away.
- **Increased database operations**: Similar to solution 1, the database still needs a uniqueness constraint, and the safer flow is still: generate candidate, insert, handle unique constraint violation, retry. The annoying bit is that retrying is not as straightforward as asking the system for another random value. Hash functions are deterministic, so the same long URL will keep producing the same short URL. If that short URL collides because we truncated the hash to N characters, we need to change the input before trying again. That can mean adding a salt, nonce, user ID, or attempt number, or increasing N. It works, but it adds more state, more database operations, and possibly more latency.
- **Deduplication is a double-edged sword**: Hear me out before calling me crazy, as I just mentioned, deduplication is a nice byproduct of using hash functions. But what if someone doesn't want deduplication? What if you don't want two users to end up with the same short URL for the same long URL? Instead, you want each one to have a different short URL. There can be many reasons for it, and the one that comes to mind immediately is that they are different users, so why should they share the same short URL? If two users submit the same long URL and you return the same short URL, who owns it? Who can delete it? Whose analytics dashboard gets the clicks? What happens if one user reports abuse and another user still depends on the link?

    Now the issue is flipped; it's easier to achieve deduplication, but it's tedious to do the opposite. You can concatenate user-specific data, such as the user ID, with the long URL, then pass the result through the hash function. If you want a different short URL for every submission, even from the same user, you can add a random nonce or salt to the input before hashing. HMAC can also help if you want the input to include a server-side secret, but it does not magically remove the need to store enough data to handle retries and collisions.
- **URL canonicalisation gets messy**: Hashing is deterministic, but URLs are annoyingly flexible. `https://example.com`, `https://example.com/`, `https://EXAMPLE.com/`, and `https://example.com?utm_source=newsletter` may or may not represent the same destination depending on your policy. If deduplication matters, you need to decide how to canonicalise URLs before hashing. Do you lowercase the host? Remove tracking parameters? Preserve query parameter order? Strip trailing slashes? Each decision has edge cases, and the edge cases will find you.

## Solution 3: Short URLs from an incrementing counter

This is as simple as it sounds. Keep track of the generated short URLs through an always-increasing counter. Each time a short URL needs to be generated, simply increment the counter and encode the value through base62. In practice, we don't even need to truncate the output of base62 encoding, as we did with the hash function solution. Assume we generated our 20 billionth URL, based on the counter value 20,000,000,000. Encoding it in base62 yields `LpVxdA` with the `0-9A-Z-a-z` alphabet; that's just 6 characters. We can generate up to approx. 56 billion 6-character-long short URLs before moving to 7 characters, and the best part is that with 7 characters, we can generate over 3.5 trillion URLs. In theory, this is actually a beautiful solution, no collisions, no database lookups to verify, the counter can even live in memory (perhaps Redis). But.... yeah, there are always some trade-offs. We don't live in a very ideal world. Let's talk about the challenges again.

### Challenges

- **Distributed systems are messy to deal with**: The solution works beautifully for a single system, but comes with issues as soon as horizontal scaling is introduced and the system becomes distributed. The primary issue is where this global counter lives, and how different parts of this distributed system synchronise with it. How to handle race conditions and concurrent access to the counter among multiple services? What about so many network requests to a central counter, which is a single point of failure under high load? You see, these issues haunt every distributed system, so one must be prepared to deal with them. One way to reduce load on the central counter is to use counter batches.

    Assume we are using Redis to store this counter. Redis commands such as `INCR` and `INCRBY` are atomic, making it a common choice for counter allocation. Each distributed service can request a batch of counter values from it. Let's say the first service claims a 1-1000 range, the second claims a 1001-2000 range, and so on. For each request, Redis increments the counter by the batch size. So if the batch size is 1000, the first `INCRBY` returns 1000 and the service owns 1-1000. The next `INCRBY` returns 2000 and the next service owns 1001-2000. Later, whichever service has consumed all the values can request a fresh batch. One problem is that batches introduce gaps. If a service claims 1-1000, uses 1-37, and then crashes, values 38-1000 should usually be considered burned. Reusing them sounds tempting, but now you have to track partial-batch state across failures, and suddenly your simple counter has grown into a small paperwork department. Most systems accept the gaps.

    Another issue with using Redis is what happens when Redis restarts, fails over, or restores from an old snapshot? Reusing old counter values would create collisions, which is exactly the thing we are trying to avoid. The counter needs persistence guarantees, backup/restore discipline, and probably a database-level unique constraint anyway as a final guardrail. Yes, even in the "no database lookups to verify" solution, the database still gets a vote.

    A single global counter can become a latency problem if services run in multiple regions. Every region has to talk to the same counter, or you need a smarter allocation strategy. Common options include pre-allocating large ranges per region, adding a region prefix, or using separate counters per shard. All of these solve one problem and introduce some new ones.
- **Predictability and enumeration came uninvited**: This is the big one if you care about privacy. If the generated short URLs are based on an incrementing counter, then nearby values are easy to guess. If I see `/abc123`, I can try `/abc124`, `/abc125`, and so on. That might reveal other people's links, and now we have a privacy problem. Public short URLs are public, sure, but "public if you know the URL" is different from "trivial to enumerate". You can reduce this by adding access controls, rate limits, abuse detection, or by encoding the counter through a reversible obfuscation step. The important bit is not to confuse obfuscation with cryptographic secrecy.
- **No natural deduplication**: Unlike the hash-based solution, a counter does not care whether the long URL has been seen before. The same long URL can get a new short URL every time. This also shows up during retries. If a user request fails after the counter has been incremented but before the response reaches the client, a retry may allocate a second short URL for the same original request. If deduplication or idempotency is required, you need a reverse lookup from long URL to short URL, an idempotency key, or some request-level deduplication mechanism. And now we are back to reverse database lookup indexes and policy decisions.
- **Business information can leak**: Sequential counters leak volume. If one short URL decodes to roughly 20 billion and another one tomorrow decodes to roughly 20.1 billion, someone can infer how many URLs were created in between. Maybe you don't care, but just politely ask your business team, and they will care very much.
- **Custom aliases and reserved paths need protection**: This is similar to the challenge in solution 1. If users can choose custom aliases, then you have to ensure generated URLs don't collide with reserved endpoints or custom aliases.

## Tiny Links, Real Trade-offs

There is no universally perfect way to generate short URLs. Random values, hashes, and counters can all work, but each one shifts complexity elsewhere: probability, state, coordination, or product policy.

My personal preference is usually a counter-based solution because the risk is easier to model. Random and truncated-hash approaches ask you to live with probabilistic collisions; counters trade that for predictable capacity planning, distributed coordination, and a few operational debts you can see coming.
