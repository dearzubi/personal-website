You must be familiar with those short, weird links, which, upon clicking, can either take you to your favourite blog (this one), or a scammy website, or some darker corner of the internet. They are typically called short URLs, where a service takes your actual URL (often called a long URL) and provides you with a shorter version. Internally, the service keeps a mapping of short URL to long URL. Therefore, when you visit that short URL, you are actually making a request to that service, which resolves that short URL to the corresponding long URL, and then redirects you to the resolved long URL. It often works so quickly that you hardly notice the redirect. Have you ever wondered how that short URL get generated? The primary observations indicate that, regardless of how it is generated, it must be unique. No two long URLs should share or point to the same short URL (often called collisions), or a giant space monster might swallow our whole solar system. How can we honour such a fragile and sacred rule? Let's explore a few solutions.


## Solution 1: Short URLs from random data/bytes

This one is extremely basic in theory. You just ask your system to generate a totally random value of a certain length, and that's your short URL. System (your machine + your environment + your favourite programming language) talks in bits and bytes. So asking for a random value of a certain length N is usually asking the system to give you N random bytes, which you can then encode in a different numeric system like decimal, hex, base62, etc. This solution is very easy to implement, but it comes with a few challenges.

### Challenges

- **Needs good system entropy**: Entropy in itself is a lengthy topic, but in simpler words, it means it's a measure of unpredictability. A high entropy system yields more unpredictable (or random) values, whereas low entropy results in the opposite. We need high entropy to minimise collisions in the generated short URLs. In such a system, collisions always occur, even if the probability is extremely low, because the system is probabilistic. There will always be a chance that the next randomly generated value is the same as the one generated some time ago.
- **Higher database lookups**: Because there is always a chance of a collision, you have to enforce a unique constraint in your database, and on each new generated value, you have to check the database to ensure it does not exist before. Or, you can try inserting the data into the database, and if a unique constraint is violated, you can retry generating a new one. This can introduce extra latency in the case of a collision.
- **Deduplication is harder**: This might not be much of an issue, but if deduplication is a goal, then it's tedious to achieve because each new generation results in a new short URL even if the long URL remains the same. The only way to achieve deduplication is to perform a reverse lookup in the database: given a long URL, find a short URL and return it. This can introduce latency (a trip to the database), and you might also need to maintain additional database index for those reverse lookups only to serve a small number of deduplication queries. However, as I said, it's only an issue if deduplication is required. Otherwise, this can be safely ignored; it's not like the world is ending on a Monday.

## Solution 2: Short URLs from hash functions

Hash functions have a special property: given arbitrary-length input data, they uniquely map it to a fixed-length output. In theory, each different input data will always map to a different fixed-length output. As you might have guessed, there is always a chance of collision simply because there is a finite set of output space, and two inputs can map to the same output, but depending upon the length of the output space, the chances are usually extremely low that we mostly ignore them. For example, a very common hash function is SHA-256. The outputs are always 256 bits long, and there can be roughly 1.16 × 10^77 possible outputs. For reference, there are an estimated 10^80 atoms in the observable universe. Imagine how much computing power is required to find a collision. Although theories like the [birthday paradox](https://en.wikipedia.org/wiki/Birthday_problem) increase the chance of finding a collision, the chances still remain extremely low for any existing practical system.

Enough chit-chat about the hash functions. Let's explore how they serve as a viable solution for our problem. It's pretty obvious, actually: we pass our long URL through a hash function and get an output we can use in our finalised short URL. You see, I did not say we use the output as a short URL; technically, we can, but we usually don't. The reason is that the output is usually long enough to defeat the purpose of the short URL in the first place. For example, if we use SHA-256, we get exactly 256 bits as output and depending on the encoding, you can get anywhere between 32 and 64 characters. Not a good way to represent a short URL, which in fact might be longer than the original long URL.

So what do we do? Well, one way is to encode the output of a hash function in base62 and then take N characters from it. Why base62 and not the popular cousin base64? Well, base64 also includes characters like `+` and `/`. Those characters usually have special meaning in a URL; `+` might represent spaces, and `/` might represent a URL segment, and both of them can mess up our short URL. base62, on the other hand, only has 26 letters [a-z], 26 letters [A-Z], and 10 digits [0-9]. Taking N characters from the base62-encoded output helps us shorten the original output, making it suitable for use as a short URL.

N is now important, as it controls the output space (the number of possible unique short URLs) and hence the likelihood of collisions. For example, if N = 8, then there are 62 ^ 8 = approx. 218 trillion possible unique outputs. A short URL with just 8 characters is still compact enough. A very nice byproduct of this solution emerges from the deterministic nature of the hash functions: same input -> same output, which means we can very easily achieve deduplication. The same long URL always produces the same short URL. Hash function solution is great, and one of the things you should try out, but it has its own challenges.

### Challenges

- **Collision control**: I know, I know. You might be thinking, if good hash functions practically have almost zero chances of collisions, then what's this fuss about? Well, remember, we encoded the output in base62 and then took N characters. That N is the root of this challenge. Set it low, you get nice short URLs, but you increase the collision chances; set it high, and you get longer short URLs but lower chances of collisions. The chances of collisions are calculated as: `n` / `|N|`. Where `n` represents the total number of short URLs generated so far, and `|N|` represents the total possible unique short URLs.

    Let's assume N = 8 and n = 10 billion. Now, the chance that a newly generated short URL will collide with an existing one is approximately 10 billion / 62^8 = approx. 0.00458%. That's about 1 collision for roughly every 22K URLs generated. You see, the risks are harmful enough for any system operating at scale. The worst part is that the collision probability keeps increasing as more and more URLs are generated, unless we increase the value of N, which lowers the probability but does not make the situation entirely go away.
- **Deduplication is a double-edged sword**: Hear me out before calling me crazy, as I just mentioned, deduplication is a nice byproduct of using hash functions. But what if someone doesn't want deduplication? What if you don't want two users to end up with the same short URL for the same long URL? Instead, you want each one to have a different short URL. There can be many reasons for it, and the one that comes to mind immediately is that they are different users, so why should they share the same short URL? Now the issue is flipped; it's easier to achieve deduplication, but it's tedious to do the opposite. You probably want to use HMAC with a random salt to introduce some unpredictability in the hash function's output. Or perhaps concatenate user-specific data, such as the user ID, with the long URL, then pass the result through the hash function.
- **Increased database lookups**: Similar to solution 1, we now need to enforce a uniqueness constraint in the database, and then for each generated short URL, we need to check if it exists or not. If so, we have to regenerate a new one. The regeneration part, however, is not as straightforward as in the case of solution 1, where you would just ask the system to hand you out another random set of bytes. Here, the deterministic nature of hash functions and the long URL-to-short URL conversion function will always result in the same short URL, because now the primary ingredient for collision is not the hash function itself, but the latter part, where we truncate the output to N characters. With N, we have only a limited output space, and collisions are unavoidable at scale. Therefore, again, we need a way to introduce randomness using a salt and HMAC, adding complexity and possibly latency.

## Solution 3: Short URLs from an incrementing counter

This is as simple as it sounds. Keep track of the generated short URLs through an always-increasing counter. Each time a short URL needs to be generated, simply increment the counter and encode the value through base62. In practice, we don't even need to truncate the output of base62 encoding, as we did with the hash function solution. Assume we generated our 20th billionth URL, based on the counter value 20,000,000,000. Encoding it in base62 yields `LpVxdA`; that's just 6 characters. We can generate up to approx. 56 billion 6-character-long short URLs before moving to 7 characters, and the best part is that with 7 characters, we can generate over 3.5 trillion URLs. This is actually a beautiful solution, no collisions, no database lookups to verify, the counter can even live in memory (perhaps Redis). But.... yeah, there are always some trade-offs. We don't live in a very ideal world. Let's talk about the challenges again.

### Challenges

- **Distributed system issues**: The solution works beautifully for a single system, but comes with issues as soon as horizontal scaling is introduced and the system becomes distributed. The primary issue is where this global counter lives, and how different parts of this distributed system synchronise with it. How to handle race conditions and concurrent access to the counter among multiple services? What about so many network requests to a central counter, which is a single point of failure under high load? You see, these issues haunt every distributed system, so one must be prepared to deal with them. One way to reduce load on the central counter is to use counter batches. Assume we are using Redis to store this counter. Redis is single-threaded and proviedes great atomicity guarantees. Each distributed service can request a batch of counter values from it. Let's say the first service claims a 1-1000 range, the second claims a 1001-2000 range, and so on. For each request, Redis simply increments the counter value by the range end value, and because of atomic operations, there can be no race conditions. Later, whichever service has consumed all the values can request a fresh batch.
